Time to bring alive the Documentum store connector

Hi there AppWorks fans,

Welcome to a new installment of AppWorks tips.

Today is the day we finally grab the time to make a deep dive on the connection between our beloved AppWorks platform, and the Documentum platform. For this we will use the ‚ÄòDocument Store Connector‚Äô which is delivered with the AppWorks platform. I must admit, I played already a couple of times with the connector, but it never ‚Äòclicked‚Äô together all by myself (for some strange reasons!?). From an insider knowledge tip I was told to really make it another try as the connection is made much simpler. Where I learned to use 2 OTDS tenants with ‚ÄòAuto-Provisioning of Accounts‚Äô together with OAuth handlers it should now be much easier with just 1 OTDS tenant and an OAuth client ID/secret setting‚Ä¶Let‚Äôs dive into this mystery of information and see how far we can get‚Ä¶

Let get right into it‚Ä¶

Before we continue we‚Äôll make a list of things we need to have up and running before we can start this post. It might be a pretty technical list, so if it doesn‚Äôt ring a bell I would recommend some research (it‚Äôs all explained on this blog site‚Ä¶have a search on the topics):

  • AppWorks platform (TomEE) connected via OTDS (‚Äòappworks‚Äô partition)
    • URL (via OTDS): http://192.168.56.107:8080/home/appworks_tips/?authID=OTDS
    • Account: awdev@appworks (with the ‚ÄòDeveloper‚Äô role applied!)
  • Documentum (Connection broker, repository and Java Method Server)
    • URL JMS: http://192.168.56.107:9060/DmMethods/servlet/DoMethod (default 9080, but we moved it!)
    • You can also check this URL: http://192.168.56.107:9060/dmotdsrest
    • As well as this URL: http://192.168.56.107:9060/OTDSAuthentication/servlet/authenticate
  • Client for Documentum (can be Documentum Administrator on Tomcat, bus this post will also deploy Documentum ReST as valid client)
    • URL: http://localhost:8080/da (I run it outside my VM as we don‚Äôt need it continually!)
    • Account: sysadmin
  • OTDS (1 tenant where OpenDJ needs to start manually‚Ä¶Just for me, so I don‚Äôt forget!) üòé
    • URL: http://192.168.56.107:8181/otds-admin/
    • Account: otadmin@otds.admin

Documentum ReST deployment

The document store connector we will bring to life on our AppWorks platform has a dependency with the exposed ReST layer of the Documentum platform. By default, this is not installed, so we first need to download it. I have Documentum 21.2 up and running and will also use the same version for the ReST client (dctm-rest.war).

Next step is to update some files in this WAR file. I open the file with 7-Zip so I‚Äôm able to update these 2 files:

\dctm-rest.war\WEB-INF\classes\dfc.properties

This makes sure our ReST client ‚Äòknows‚Äô where our Documentum repository is running.

\dctm-rest.war\WEB-INF\classes\rest-api-runtime.properties

This makes sure our ReST client ‚Äòknows‚Äô how to authenticate via OTDS.

FYI: The document store connector makes sure we use the same OTDS authenticated user account for the AppWorks platform as well as the Documentum platform! That‚Äôs technically a better solution for security than the use of a so-called ‚Äòservice-account‚Äô!

Our WAR file is prepared, time to deploy it on our JMS. This is just an upload action to the VM in this location: /opt/documentum/tomcat9.0.43/webapps/

Tomcat will ‚Äòexplode‚Äô the WAR file to a ‚Äòwebapps‚Äô directory. After a minute you should be able to access these URLs:

  • http://192.168.56.107:9060/dctm-rest

  • http://192.168.56.107:9060/dctm-rest/repositories

    Pro-TIP: Add .json in the URL: http://192.168.56.107:9060/dctm-rest/repositories.json and make sure you have a ‚ÄòJSON formatter‚Äô plugin installed in your Chrome browser. This makes it much easier to ‚Äòbrowse‚Äô through the JSON ‚Äúhref‚Äù URLs!

  • http://192.168.56.107:9060/dctm-rest/repositories/aw_repo.json (will give an ‚ÄòAuthentication failed‚Äô for now as we didn‚Äôt connect OTDS (yet!))

    If you already want to browse around, you need to comment those 3 lines in the ‚Äòrest-api-runtime.properties‚Äô and do a restart of the JMS systemclt restart dm_jms. After restart, you will be able to authenticate via the non-OTDS account ‚Äòsysadmin‚Äô.

Configure OTDS

Now, we do exactly what we ‚Äúnormally‚Äù would do for our AppWorks platform to authenticate with OTDS. We create a new resource (this time a resource for Documentum), and we add the already excising ‚Äòappworks‚Äô partition as access role. So, in the end we have 1 partition with users/groups which is used in 2 resources (one for Appworks itself and one for Documentum)

Let‚Äôs dive into OTDS http://192.168.56.107:8181/otds-admin/ and login with ‚Äòotadmin@otds.admin‚Äô. Go to the ‚ÄòResources‚Äô category and ‚ÄòAdd‚Äô a new resource:

Use this information:

  • General tab

    • Resource Name: dctm_resource

  • Synchronization tab:

    • Mark ‚ÄúUser and group synchronization‚Äù
    • Synchronization connector: REST (Generic)

  • Connection information:

    • http://192.168.56.107:9060/dmotdsrest

    • Username: aw_repo\sysadmin

  • User Attribute Mappings tab:

    • Set the client_capability = 2 for the users (so, they will be ‚Äòcontributor‚Äô in the repository of Documentum)

Now do a save of the resource and copy the ID: dd83d441-6a52-42e6-b7bd-177ce0bea4e0

!!DON‚ÄôT HIT THE VERIFY ACTIVATION BUTTON!!

Normally we activate a resource via the AppWorks platform when we configure OTDS in the ‚ÄòSecurity Administration‚Äô artifact‚Ä¶remember?

Only from a Documentum perspective we don‚Äôt have this option (yet!?) via a nice UI. For Documentum, we need to make a manual HTTP POST to this URL (where the ID is our just copied resource ID!):

http://192.168.56.107:8181/otdsws/rest/resources/dd83d441-6a52-42e6-b7bd-177ce0bea4e0/activate

We can use a tool like Postman to make this call, but I downloaded CURL for Windows so I‚Äôm able to execute a prompt command like this:

curl -X POST http://192.168.56.107:8181/otdsws/rest/resources/dd83d441-6a52-42e6-b7bd-177ce0bea4e0/activate

You will get a response in JSON format. Save that retrieved ‚ÄòsecretKey‚Äô:

Optional: In OTDS you can now also ‚ÄòVerify‚Äô the activation status for this activated resource!

Test Documentum authentication via OTDS

Finally!‚Ä¶Time to test our first authentication!

Connect to IAPI: /opt/documentum/product/21.2/bin/iapi aw_repo with sysadmin superuser account

Now give this API statement connect,aw_repo,awdev@appworks,dm_otds_password=admin and we should be able to start a new session for this user with the Documentum repository‚Ä¶GREAT!!

I tried this call before we configured JMS with OTDS and this call is NOT working until JMS is prepared (I thought it would work already as the awdev.json already gave result from the ReST call earlier for some strange reason, but that‚Äôs probably updates from my side!?)

Now for the OAuth authentication: connect,aw_repo,awdev@appworks,dm_otds_oauth=<access_token>

The access token can be retrieved via this URL call in the browser: http://192.168.56.107:8181/otdsws/oauth2/auth?response_type=token&client_id=dctm_oauth_client&client_secret=QZxvH2l81Fp6rxE7ho23vY8xVYTdz15Q&redirect_uri=http://192.168.56.107:8181/otdsws/login

Make sure to pass the correct parameter values for the ‚Äòclient_id‚Äô and ‚Äòclient_secret‚Äô, but you have that information available now!

The URL will redirect you to the OTDS authentication screen where we need to login with our ‚Äòawdev@appworks‚Äô credentials. When done we will retrieve an ‚Äòaccess_token‚Äô as parameter in the URL which can be used in the API connect call‚Ä¶With a satisfying API session result: ‚Äòs1‚Äô

Hooray‚Ä¶Call yourself PRO when you made it this far! 🤓
